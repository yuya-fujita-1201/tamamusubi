## 1. フラグ管理

### 1.1 フラグID の正式確定

[EVENT_LIST §0.3](../../10-lists/EVENT_LIST.md) に記載された「前提フラグ（暫定記法）」を、本書で **`flg_###`（3桁ゼロ詰め）** に正式昇格させる。これ以降の全設計書・実装ファイルは `flg_###` 形式を使用すること。

### 1.2 フラグ種別

| 種別 | 説明 | ID帯（暫定） | 例 |
|---|---|---|---|
| グローバル進行フラグ | 全マップ共通。幕・依代・交通の進行状態を保持 | flg_001〜099 | flg_001 序幕完 |
| ローカルフラグ | 特定マップ・シーンに紐づく一度きりの状態 | flg_100〜299 | flg_100 ev_048完了（里の子たけ） |
| カウンタ | 累積値を持つ（依代数・祓い数・収集数） | flg_300〜349 | flg_300 依代灯数(0〜8) |
| アイテム所持判定 | 特定アイテムの所持可否をトリガー条件として参照 | flg_350〜399 | flg_350 船入手 |

> アイテム所持判定は ITEM_LIST の itm_### と1対1で対応させる。アイテム取得時に自動で関連フラグを立てる実装とする。

### 1.3 命名規則

```
flg_{NNN}_{スラッグ}
  NNN  : 3桁ゼロ詰め通し番号
  スラッグ: 英数・アンダースコアのみ。省略可（グローバル進行フラグは省略する）
```

例: `flg_001`（序幕完）、`flg_015_r3`（R3主ボス撃破）、`flg_300`（依代灯数カウンタ）

## 2. トリガー種別

イベントを発火させる条件の分類と、実装上の規則を定める。

| トリガー種別 | 説明 | 一度きり or 反復 | 優先度 |
|---|---|---|---|
| 接触 | 特定オブジェクト・NPCに主人公が重なる | 一度きり（メイン）／反復（商店・宿） | 中 |
| 会話 | NPCに話しかける（Aボタン/タップ） | 反復（内容はフラグで変化） | 中 |
| アイテム取得 | 特定アイテムを入手する | 一度きり | 中 |
| 敵撃破 | 特定の敵またはボスを撃破する | 一度きり（ボス）／反復（雑魚） | 高 |
| エリア進入 | 特定マップに初めて入る | 一度きり（初回）／反復（再入）で分岐 | 低 |
| フラグ条件成立 | 指定の flg_### が立っている状態でトリガー地点に到達 | 一度きり | 最高 |

### 2.1 優先度の運用ルール

- 同時に複数のトリガーが成立した場合、優先度の高い順に処理する。
- 同優先度が競合した場合は、**メインイベント > サブイベント > 環境語り** の順で発火させる。
- 一度きりトリガーは発火後にフラグ（flg_1xx台）を立て、二重発火を防ぐ。

### 2.2 「会話を必須トリガーにしない」原則

[Story.md §6 審査改善1](../../../Story.md) および [EVENT_LIST §0.1](../../10-lists/EVENT_LIST.md) の不変条件に従い、**メインシナリオの進行ゲートに NPC 会話を使用しない**。進行ゲートは必ず「依代灯数（flg_300）」または「ボス撃破フラグ（flg_010〜015）」で構成する。NPC は情報・依頼・スキルを与えるのみ。

## 3. イベント3分類の運用

[EVENT_LIST §0.2](../../10-lists/EVENT_LIST.md) の分類を本書の実装ルールとして確定する。

| 分類 | ID帯 | 件数 | 役割 | 制作ルール |
|---|---|---|---|---|
| メイン | ev_001〜040 | 40 | 幕進行の核。依代/ボス/層越え/真相 | 必ず前後のフラグ変化を明記。一度きり。無台詞シーンにも ev_### を割り当てる |
| サブ | ev_041〜110 | 70 | 地域の依頼・寄り道・スキル習得 | 前提フラグは任意。完了フラグ flg_1xx を立て重複受注を防ぐ |
| 環境語り | ev_111〜150 | 40 | 遺物・痕跡・足跡・碑。説明テキストなし | トリガーは「接触」または「エリア進入」。台詞・説明文は一切付けない |

### 3.1 メインイベントの制作ルール詳細

1. **幕ごとの番号帯**: 序幕 ev_001〜005 / 第一幕 ev_006〜027 / 第二幕 ev_028〜033 / 第三幕 ev_034〜038 / 終幕 ev_039〜040。番号帯を跨ぐ移動は禁止。
2. **無台詞シーン（15秒演出）も ev_### を持つ**: ev_008・ev_012・ev_016・ev_020・ev_023・ev_026 など。フラグ変化と報酬付与はこのイベントが発火した瞬間に処理する。
3. **祓い演出の段階変化**（[Story.md §7 審査改善2](../../../Story.md)）: 序盤(R1〜R2) = 砕け散る / 中盤(R3〜R6) = わずかに上へ / 常世以降(R8〜R10) = 昇る。該当イベントのデータに `purify_stage` フィールドで記録。

### 3.2 環境語りの制作ルール詳細

- **密度の上限**: 3画面スクロール（約1024px換算 × 3 = 3072px相当）に1件以下。詳細は §4 を参照。
- トリガー発火後は「調べた」フラグを立て、同じ内容が繰り返し出ないようにする（ただし「灯した依代を再訪すると光が増す」ev_149 など視覚変化イベントは反復可）。

## 4. 環境語り

### 4.1 種別と実装形式

環境語りは台詞・説明文・「！」マークを出さない。以下の4形式で世界観を伝える。

| 形式 | 典型例 | 実装方法 |
|---|---|---|
| 拾い読み | 絵馬・落書き・石碑 | 接触で短い「テキスト画像」を表示（UIウィンドウではなく世界内の物体として） |
| 碑文 | 封印を讃える碑・削られた傷跡 | ev_124/ev_143。同じフォントを使いながら色・破損度を変えて年代を示す |
| 手控え帳 | itm_051〜058 の各段 | アイテムメニューから読む長文テキスト。プレイヤーが能動的に開かなければ読まない |
| 独り言 / 痕跡 | 補修跡・凍った勾玉の欠片・道具 | 接触SEのみ。テキストなし。視覚と音で完結する |

### 4.2 密度ルール（秘密の密度）

- **上限: 3画面に1件**。環境語りが密集しすぎると「語りかけてくる世界」ではなく「説明しすぎる世界」になる。
- ダンジョン内は密度を高めてよい（移動距離が短いため実効密度は変わらない）。
- 野外マップの環境語りは、最寄りの町NPCの会話とテーマが重複しないよう設計する（例: ev_115 案山子の列 と npc_014 与平の会話は「里の弱り」という同テーマだが、伝え方を視覚/言語で分ける）。

### 4.3 二層設計

気づかなくてもクリア可能・気づくと世界が違って見える二層構造を維持する。

- **表層（誰でも気づく）**: 依代の光・ボスの演出・手控え帳アイテム
- **深層（探索者向け）**: ev_114 子供の落書き「神様は静かになっただけ」・ev_124 碑文と ev_143 傷跡の対応・ev_138 どこへも続かない足跡

## 5. 会話・分岐

### 5.1 NPC_LIST との接続

NPC の会話は以下の3状態を持つ。

| 状態 | 条件 | 会話内容 |
|---|---|---|
| 初回 | 対応 flg が未成立 | イベント前の通常台詞（1〜3行） |
| 完了後 | 対応 flg が成立 | 別の短い台詞または無言（反復防止） |
| 特定フラグ参照 | 別の flg が成立している場合 | 追加情報・深い語り |

### 5.2 会話の構造方針

- **主人公は選択肢に出ない**。主人公が聞き役で、NPC が喋る。選択肢は「もう一度聞く」「やめる」の2択のみ。
- **1人3行以内**（[Story.md §7 配分ルール](../../../Story.md)）。過剰な説明・世界設定の解説は禁止。
- 真相・世界の呼吸・天鎖についてはいかなるNPCも直接説明しない。常世の巫（npc_004）の台詞は五七五の短詩のみ（[Story.md §2](../../../Story.md)）。

### 5.3 分岐フラグ

複数の結果を持つ会話分岐は使わない（主人公は寡黙で選択肢を持たないため）。**フラグによる会話内容の切り替えはある**が、プレイヤーが選択で分岐する構造は採用しない。

## 6. セーブ／進行データ構造方針

### 6.1 保存対象

| データ種別 | 具体内容 | 形式 |
|---|---|---|
| フラグ集合 | flg_001〜399 の真偽値またはカウンタ値 | ビット配列 + 整数配列 |
| アイテム所持 | itm_### の所持数・入手済みフラグ | 整数配列 |
| 現在位置 | map_id + 座標 (x, y) | 文字列 + 整数ペア |
| 図鑑 | 妖の図譜 itm_059 の発見フラグ | ビット配列 |
| 手控え帳 | itm_051〜058 の入手フラグ + 閲覧済みフラグ | ビット配列 |
| スキル習得 | skl_### の習得フラグ | ビット配列 |
| 釣り手帖 | itm_061 の釣魚フラグ | ビット配列 |
| 拓本帖 | itm_060 の収集フラグ | ビット配列 |

### 6.2 オートセーブ

- **依代（鳥居・祠）に触れたとき**に自動セーブ。
- **エリアをまたいだとき**（マップ遷移）にオートセーブ。
- ボス戦前の最終セーブポイントは依代で代替する（専用チェックポイントは設けない）。

### 6.3 依代 = セーブ拠点

依代はゲーム世界の文脈上「世界の流れの灯台」（[Story.md §5.1](../../../Story.md)）であり、システム上も「HP回復＋セーブ」の拠点を兼ねる。宿（npc_011 おりつ等）も休憩でHP全回復を提供するが、宿はオートセーブではなく手動セーブの窓口とする。

### 6.4 データ容量方針

フラグ 400本分 + カウンタ 50本分 + アイテム配列で概算 < 4KB/スロット。スロットは3つ用意（手動 × 2 + オート × 1）。
