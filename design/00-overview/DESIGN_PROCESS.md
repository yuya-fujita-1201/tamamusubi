## 1. 工程モデル（一覧 → 基本 → 詳細）

SI開発の設計工程に対応する3層。**上から具体化し、下から整合**を取る。

```
① 一覧図 (10-lists)   …… 全要素を1行/1セルで網羅。数量目標を満たすマスターリスト
        │ ID を確定して受け渡す
        ▼
② 基本設計 (20-basic) …… 地域/システム単位で「完成像」を文章設計。一覧の行を束ねる
        │ 個別要素を選んで深掘り
        ▼
③ 詳細設計 (30-detail) …… 実装に落とせる粒度。マップは map-quality パイプラインへ接続
```

- **一覧に無いIDを詳細設計してはならない**（背骨を貫通させるため）。
- 詳細設計で新要素が必要になったら、**まず一覧に追加** → 基本設計に反映 → 詳細を書く。
- 数量目標（[SCOPE_AND_SCALE.md](./SCOPE_AND_SCALE.md)）は一覧層で管理し、基本・詳細はその内訳を具体化する。

## 2. ID 命名規則（相互参照のキー）

一覧↔基本↔詳細は **ID** で結ぶ。IDは原則 ASCII 小文字＋数字＋アンダースコア（snake_case）。

| 要素 | ID形式 | 例 | 備考 |
|---|---|---|---|

## 3. ファイル命名・節構成

- ファイル名は ID または ID+説明（例 `R1_satoyama.md` / `W1_haraitou.md` / `MAP_DETAIL_TEMPLATE.md`）。
- **新規文書は必ず `templates/` の対応テンプレからコピーして起こす**。節の追加は可、削除は不可（空でも見出しを残す）。
- 機械可読データは `10-lists/data/*.json`。snake_case キー、先頭に `"schemaVersion"` を付与し、Markdown表のミラーとする。

## 4. 版管理（メンテ体制の核）

### 4.1 ステータス区分

| 状態 | 意味 |
|---|---|
| `未着手` | ファイル未作成、または見出しのみ |
| `雛形` | テンプレ流し込み済み・一部記入（骨格はある） |
| `第一版完` | 当該層の第一版として網羅・整合済み（検収基準を満たす） |
| `更新中` | 第一版完了後に改訂作業中 |

### 4.2 ルール

1. 文書を**新規作成・更新したら、同じ作業の中で [REVISION_LOG.md](../REVISION_LOG.md) の該当行を更新**する。
2. 各文書の先頭に `> 版: vX.Y / 最終更新: YYYY-MM-DD / 状態: …` の1行を置く。
3. 上位文書（PLAN/Story/Systems）から**逸脱・拡張・解釈**した箇所は REVISION_LOG の「整合メモ」に1行残す。
4. **書かれていないことは「やっていない」とみなす**（グローバル方針）。チャットだけで完結させない。

## 5. Definition of Done（第一版の検収基準）

コードは動かさないため**ドキュメント整合性**で検収する（PLAN由来の数量目標は [SCOPE_AND_SCALE.md](./SCOPE_AND_SCALE.md)）。

| # | 検収項目 | 目標 |
|---|---|---|
| 1 | README のインデックスから全文書へ辿れる（リンク切れ無し） | 100% |
| 2 | MAP_LIST 行数 | ≈303（v1.2／旧≈250） |
| 3 | WEAPON_LIST 件数（系統×段階） | 80（8×10） |
| 4 | ITEM_LIST 件数 | ≈115（v1.1／旧100） |
| 5 | 地域基本設計 | 10本（v1.2／旧8本） |
| 6 | システム基本設計 | 7本 |
| 7 | MAP_LIST の全 map_id が対応地域の基本設計に登場（背骨貫通） | 未解決参照 0 |
| 8 | satoyama 全マップに 30-detail 詳細があり map_goal/heightmap 入力節を含む | 100% |
| 9 | WEAPON 詳細が8系統網羅・各10段階が WEAPON_LIST と一致 | 一致 |
| 10 | REVISION_LOG に全文書が1行ずつ状態付きで記載 | 100% |

マップ詳細設計のさらなる合格基準は `../quality/MAP_DEFINITION_OF_DONE.md`（総合スコア80点以上 / ERROR 0 / 重大WARNING 3件以下）に従う。

## 6. 作成順序（波状）

大量文書のため、基盤を自分で固め、一覧→基本→詳細を波状に展開する。

| Wave | 内容 | 担当 |
|---|---|---|
| 0 | 骨格・README・REVISION_LOG・GLOSSARY・DESIGN_PROCESS・SCOPE・RESEARCH・PRODUCT_OVERVIEW・templates 5種・REGION_LIST | 自分 |
| 1 | 一覧（MAP/WEAPON/ITEM/ENEMY/BOSS/NPC/EVENT/ACCESSORY/SKILL）＋ data/*.json | 並列 |
| 2 | 基本設計（地域10＋システム7） | 並列 |
| 3 | 詳細設計（satoyama全マップ＋武器8系統＋序盤イベント） | 並列 |
| 4 | 整合パス（ID相互参照・行数・不整合チェック）→ REVISION_LOG 確定 | 自分＋レビュー |

> 各波は前波の出力（特に REGION_LIST と各一覧のID）を入力にする。
